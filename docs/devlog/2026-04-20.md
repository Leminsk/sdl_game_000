## Move camera from the minimap:
Well we had a way to translate where the camera is to where it should be displayed on the minimap. Now we need kind of need to do the reverse, based on where on the minimap we click, where in the *world* would the camera be after clicking?  

Before calculating all those neat coordinates for the camera, we need to be aware that the "camera" is not really an object. Different from all entities that exist in the world, the "camera" is more of a semi-loose collection of variables which describe where the camera is and its zoom level. But did you catch that little detail? Those are distinct *independent* variables, so no matter how small or large a zoom is, in this game engine, it ***does not change the position of the camera***. All that time where the player is zooming in and out, the camera position is not changing.  
"That's obvious" you might say. But when you're used to having all of your game entities in the world with a `TransformComponent` with property `position` which indicates the "top-left" corner of the rectangle representing the entity, you can get tripped up by the camera rectangle crop changing sizes and thus its apparent top-left corner changing as well, but it's true position stays unchanged. That used to trip me up so much that I just changed the variable came to `camera_diff` to remind me that it does not work like other positions.

```cpp
static Vector2D camera_diff; // how much the camera has been moved from the origin
static float camera_zoom; // level of zoom (x0.5, x0.8, x1.0, x1.2, x1.5 etc)
```  
Now on to the math. So the player clicks on the minimap. Now what? We need to set the global `Game::camera_diff` to a value that represents where that camera would be if it were *centered* around the point clicked on the minimap. So we need to convert from minimap coordinates to world coordinates. Well that's easy enough. We just have to do the reverse of what `convertWorldToMinimap()` does. Here's what we need:
```cpp
Vector2D convertMinimapToWorld(float x, float y) {
    return {
        (x - this->map_rect.x) / this->minimap_proportion_width,
        (y - this->map_rect.y) / this->minimap_proportion_height
    };
}
```
We don't even need to bother with inversing the clamp because we're only dealing with clicks on the minimap. But we can't just call the function on the click coordinate because that would move the camera to the position of the mouse click on the world, and that would mean that the camera would point down and right of where we want. Remember, I want it *centered* on the cursor click.  
So we need to translate the position by half of the width of the camera crop on X and by half of the height of the camera crop on Y. Something like this:
```cpp
Game::camera_diff = this->convertMinimapToWorld(bx - half_width, by - half_height);
```  
But here's the actual hard part about this camera manipulation thing. Remember how the camera position and zoom are independent from each other? That means that, in a way, the camera is always on x1 zoom level. And the zoom that we see happening in the game is more akin to expanding and shrinking the universe while the camera "object" (not an object) remains fixed in place. So at first one might think that we could calculate `half_width` and `half_height` by doing this:

```cpp
Vector2D world_top_left     = convertScreenToWorld({ 0.0f, 0.0f });
Vector2D world_bottom_right = convertScreenToWorld({ static_cast<float>(Game::SCREEN_WIDTH), static_cast<float>(Game::SCREEN_HEIGHT) });
Vector2D half_vector = { // half_width and half_height are here
    (world_bottom_right.x - world_top_left.x) / 2.0f, 
    (world_bottom_right.y - world_top_left.y) / 2.0f 
};
Vector2D world_button_pos = this->convertMinimapToWorld(bx, by);
Game::camera_diff = world_button_pos - half_vector;
```
But that is **WRONG**. Yes, I have to calculate the `half_width` and `half_height` by taking half the distance between the points, but if I do that based on the corners of the screen alone, they I won't get the true corners of the actual camera, I'll get the corners of the *zoomed* camera. So I need to scale back from whatever zoom level we're currently back to level "x1" because `Game::camera_diff` *never* gets scaled, but is always used as the reference for "where the camera is".
So the actual implementation is as follows:  
```cpp
Vector2D world_top_left     = convertScreenToWorld({ 0.0f, 0.0f });
Vector2D world_bottom_right = convertScreenToWorld({ static_cast<float>(Game::SCREEN_WIDTH), static_cast<float>(Game::SCREEN_HEIGHT) });
Vector2D half_vector = { 
    (world_bottom_right.x - world_top_left.x) / 2.0f, 
    (world_bottom_right.y - world_top_left.y) / 2.0f 
};
// camera_diff is always at x1.0 zoom scale, scale back the difference by the current zoom
half_vector = half_vector * Game::camera_zoom;
Vector2D world_button_pos = this->convertMinimapToWorld(bx, by);
Game::camera_diff = world_button_pos - half_vector;
```  
*Now* we can put that in a method for `MapThumbnailComponent`, and call it whenever we get a mouse left click on our scene (and treat it to not bother claculating anything if the click is outside the minimap). And then we get this:

## Move unit from the minimap
With camera movement enabled for the minimap, we now have to consider "How do I tell the unit to move to where I clicked on the minimap?". Well, good news! We have already taken care of how to convert minimap coords to world coords. And units already have a method `moveToPoint()` which takes a world position and then deals with finding the path from the unit to that point and starts a routine of walking to that point. So all that we need is the world coords of wherever we clicked on the minimap, which we now get for free from out previous `convertMinimapToWorld()`.  

We check if we got a *righ* click on the minimap, and if we did, we overwrite the value of what the scene thinks we had clicked to being the world equivalent based on the minimap, instead of being "behind" the minimap. Which looks like this:
