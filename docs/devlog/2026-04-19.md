# 2026 April 19 - A minimap with units and buildings
Last time I said that in the future I would try to work out how to track units on the minimap. Well the future is now.  

But first, let's talk about something that I had not noticed. If you pay close attention to the GIF from the last entry, you might notice how the pink rectangle doesn't *quite* match where the camera is pointing at. Here's a frame from that GIF to better explain it:  
<img src="/sdl_game_000/assets/images/2026-04-19/janky_minimap_camera.png" alt="It's just a tiny bit off" style="max-width: 100%;">  

Look closely to the lower right side of the pink rectangle on the minimap. See how it just barely touches the grey tile? That means that we should *barely* see the grey tile sprite, which is the concentric big rock formation seen in a X shape in this image too. But instead we get to see it protruding from the side on the main view of the game. Another detail is that if we look to the lower *left* side of the pink rectangle on the minimap, we can see that only part of the bottom left tile of the camera rectangle should be visible, however, it  shoes *most* of the tile.  

What does this all mean? It means that the camera rectangle is being drawn slightly off from where it should be. I only noticed this when I was first trying to render the units on the minimap and noticed that their positions were consistently off from where they should have been drawn.  

That of course means that I was doing something wrong when converting from "world" coordinates to "minimap" coordinates. In the last code snippet I shared there is an error in by how much I need to translate the coordinates. I was wrongly using `origin_x` and `origin_y` as the translation vector. But those values are the position of the whole minimap *including the border*. After facepalming and just changing the the vector, I then decided to also include all that into a function to help me later on. So here's our now correct function on how to convert world coordinates into minimap coordinates:  
```cpp
SDL_FPoint convertWorldToMinimap(Vector2D& world_pos) {
    return {
        std::clamp(
            (world_pos.x * this->minimap_proportion_width) + this->map_rect.x, 
            this->map_rect.x, 
            this->map_rect.x + this->map_rect.w
        ),
        std::clamp(
            (world_pos.y * this->minimap_proportion_height) + this->map_rect.y, 
            this->map_rect.y, 
            this->map_rect.y + this->map_rect.h
        )
    };
}
```  
I'm returning it as a `SDL_FPoint` because it's just conveninient in my use case, but it could easily have been another `Vector2D` function much like a bunch of others that I have.  

Now, back to actually trying to draw units on the minimap. Thankfully there already exists an `std::vector<Entity*>` for units being used by the scene which creates the minimap object. So I just included that in the constructor, passed it as a pointer, and then now the minimap has access to all of the units it needs to draw. Firstly I just hard-coded everything to see if my reasoning was decent. Then I simplified it into another helper function for the minimap:  
```cpp
void drawEntityToMinimap(Entity* e) {
    TransformComponent& e_transform = e->getComponent<TransformComponent>();
    SpriteComponent& e_sprite = e->getComponent<SpriteComponent>();
    SDL_FPoint minimap_pos = this->convertWorldToMinimap(e_transform.position);
    SDL_FRect e_draw_rect = { 
        minimap_pos.x, minimap_pos.y, 
        e_transform.width * this->minimap_proportion_width, 
        e_transform.height * this->minimap_proportion_height 
    };
    TextureManager::Draw(e_sprite.texture, NULL, &e_draw_rect, e_sprite.rotation, e_sprite.spriteFlip, e_sprite.color_modulation);
}
```  
What this function does is access the `TransformComponent` (has position and dimensions) and the `SpriteComponent` (texture and other rendering logic) of whatever `Entity` pointer has been passed, and use that information to calculate where on the minimap that would be (convert to minimap), what size it would be (`e_draw_rect`), what to actually draw (`e_sprite.texture`) and how to draw it (the other arguments for `TextureManager::Draw()`).  

At this stage we don't have fog of war, and we're drawing the whole world minified on the minimap, which means that there's no need to check which units need to be drawn or not, since we're representing the *entire* game world where the units can be. So no need `if`s for AABB collision / checking for out of bounds units; all of the get drawn as mini roombas.

This worked pretty well. So much so that it is the *exact same logic* for how to draw the buildings (hexagons) which are also already in another `std::vector<Entity*>` managed by the same scene! I just pass by reference both vectors to the constructor, and call the same `drawEntityToMinimap()` on all Entities in the minimap's `draw()` method, then lastly we draw the camera's pink rectangle.
